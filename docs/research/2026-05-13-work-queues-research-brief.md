# AI Agent Work Queues — Research Brief (May 2026)

> Research input that informed `docs/superpowers/specs/2026-05-13-work-queues-design.md`. Generated by web search across 2025-2026 sources.

## Systems

- **Temporal** (Go core; TS/Python/Java/Go SDKs; MIT). Durable-execution incumbent. Feb 2026 Series D ($300M @ $5B), 9.1T lifetime executions, 1.86T from AI-native customers. Replay 2026 announced **serverless workers** and **workflow streams**. GA integrations with OpenAI Agents SDK (Mar 2026) and Vercel AI SDK. Storage: pluggable (Cassandra/MySQL/Postgres). Heavy operationally; not Postgres-native.
- **Inngest** (TS/Go; source-available + cloud). Event-driven step functions with durable resume, suspend-for-human, agent kit. Strong fit for TS stacks. Backed by its own datastore on cloud; OSS dev server uses SQLite/Postgres.
- **Trigger.dev** (TS; Apache-2.0; v4 GA late-2025). Checkpoint/restore tasks, queues, idempotency keys, warm starts 100–300ms. Postgres + object storage. Self-hostable.
- **Hatchet** (Go core, multi-SDK; MIT; v1 launched Mar 2025 on Postgres). Postgres-native task orchestration; multi-tenant queues are a marketed differentiator (>1B tasks/month). Closest architectural sibling to paperclip's stack.
- **river-queue** (Go; Apache-2.0; Postgres). Library, not a platform. Latest release Apr 26, 2026; River Pro adds durable periodic jobs, SQLite preview. SKIP LOCKED + COPY FROM batch insert; tens of thousands jobs/sec.
- **pg-boss** (Node/Postgres; MIT). v12.18.2 (May 2026). SKIP LOCKED, cron, DLQ, retries, pub/sub. Mature, lightweight, no separate control plane.
- **BullMQ** (Node/Redis; MIT). Flow producers, parent/child, rate limits, ~50k jobs/min on one Redis. Different storage from paperclip; useful as feature reference, not adoption candidate.
- **Restate** (Rust; BSL). Durable runtime targeting agents directly (Google ADK, Vercel AI SDK, Pydantic AI integrations in 2026). Persists every LLM call/tool/route decision.
- **LangGraph** (Python/TS; MIT; LangGraph 1.0 in 2026). Checkpointer pattern with Postgres/SQLite/CosmosDB drivers. Per-node checkpoint cost 5–15ms — invisible vs LLM latency. Strong in-process model, weaker as a fleet scheduler.
- **Mastra** (TS; Apache-2.0; 1.0 Jan 2026, 300k weekly npm). Treats agents and workflows as separate primitives — deterministic step graphs vs model-driven loops. Same TS/Vercel-AI-SDK lineage as paperclip-relevant tooling.

## Patterns

The 2025–2026 consensus is **durable execution as default**. Two mechanisms dominate: journal-replay (Temporal/Inngest/Restate) and DB checkpointing (LangGraph, Trigger). LLM/tool calls are wrapped as "activities/steps" whose result is journaled exactly once — replay never re-charges tokens. Idempotency keys are first-class on Trigger.dev and Inngest. Cron + event triggers + manual fanout coexist; human-in-the-loop is modeled as durable suspend/resume. Probabilistic retries (a re-run is *not* the same answer) push designs toward storing the *result* of an LLM step, not retrying the prompt.

## Backpressure

Per-tenant fairness is now a named feature (Hatchet's "multi-tenant queues" post). Standard pattern: **sharded queues per tenant + weighted fair queuing** round-robin draw, with token-pool / SLO-aware admission for LLM-heavy workloads (see arxiv "Token Management in Multi-Tenant AI Inference"). Cost-aware backoff means rate-limit retries should not re-bill — checkpoint *before* the LLM call, retry *after* failure classification (rate-limit vs poison). DLQ + poison detection (max-attempts → dead) is universal; pg-boss, BullMQ, river all ship it. Priority lanes are typically (queue, priority int) tuples.

## Concurrency

Best practice for per-key serialization in 2026 on Postgres is exactly what paperclip's `workspace_leases` Plan 4 already does: **partial unique index** on `(key) WHERE state='active'` plus `SELECT … FOR UPDATE SKIP LOCKED LIMIT 1` for the dequeue. Hatchet, river, pg-boss, and Inngest's OSS all converge here. Watch the MultiXact SLRU bottleneck above ~thousands of concurrent sessions (richyen.com May 2026). Lease leases (heartbeat-renewed, expirable) are preferred over advisory locks because they survive worker death without exotic recovery. Fanout is typically a workflow primitive ("parent task spawns N children"), not a queue feature.

## Observability

OpenTelemetry **messaging semconv reached Stable in 2026**. Spans: `create` / `send` / `receive` / `process` with `messaging.system`, `messaging.destination.name`, `messaging.message.id`. Standard messaging client metrics (queue depth, publish/consume duration, retry counters) are stable. Pair with **GenAI agent semconv** (`gen_ai.agent.*`, `gen_ai.operation.name`) to correlate queue spans with the LLM call inside the activity. Queue-depth, p50/p95 dequeue latency, retry rate, DLQ size are the canonical four signals.

## Paperclip integration considerations

Paperclip's existing primitives map cleanly:
- `heartbeat_runs` ↔ workflow execution row.
- `routines` ↔ cron/scheduled triggers (already done).
- `workspace_leases` ↔ per-key serialization lock (done; this is the river/Hatchet pattern).
- `heartbeat_scheduler` (30s tick) ↔ dequeue loop. Keep the tick, but have it `SKIP LOCKED LIMIT N` against a new `work_items` table partitioned by tenant for fairness.
- A new `work_items(tenant_id, queue, priority, payload, dedupe_key, available_at, attempts, state)` table with a partial unique index on `dedupe_key WHERE state IN ('queued','running')` gives idempotency without a separate Redis. Materializing into `heartbeat_runs` on dequeue (the "fanout into runs" you described) matches Hatchet's "task → run" split.
- Plug in OTel messaging semconv from day 1 — free dashboards in Grafana/Honeycomb.

## Open questions

1. Per-tenant fairness: shard table per tenant, or single table with weighted round-robin in dequeue SQL? Hatchet does the latter; simpler for self-host.
2. Replay model: full journal-replay (Temporal-style) is heavy in TS without a runtime; closer to paperclip's grain is **per-step idempotent re-execution** (Trigger/river) where the agent run is the unit, not the step.
3. Cost-aware backoff: classify provider 429 vs 5xx vs poison before retry; do we need a `retry_policy` per routine?
4. Cron-vs-event ingestion: routines already cron; do tickets/reviews arrive via webhook → enqueue, or via polling plugin → enqueue? Both, probably.
5. Do we expose the queue as a plugin extension point (custom dequeue strategy) or keep it core?
