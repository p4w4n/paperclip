# AI Agent Artifacts and Work Products — Research Brief (May 2026)

> Research input that informed `docs/superpowers/specs/2026-05-13-artifacts-work-products-design.md`. Generated by web search across 2025-2026 sources.

## Systems

**Claude Artifacts** (Anthropic): supports code, SVG, Mermaid, React, HTML, Markdown, .docx/.pptx/.xlsx/.pdf, plus "Live Artifacts" (April 2026) — dashboards bound to live data sources. Auto-versioning per revision, 90-day retention on shared artifacts (extended from 30 in March 2026). MCP-bound interactive apps on Pro/Team/Enterprise. Closed source, hosted.

**OpenAI Canvas**: side-by-side editor with structured shortcuts (review, comment, fix, log). Hard truncation on long generations; no mobile in 2026. Closed.

**Vercel v0**: best-in-class UI generation, one-click deploy via Vercel preview URLs. Closed/proprietary, no self-host.

**Bolt.new (StackBlitz)**: artifacts are *structured LLM outputs* parsed by `EnhancedStreamingMessageParser` and dispatched to an `ActionRunner` (`addArtifact`/`addAction`/`runAction`). Runs entirely in WebContainer (browser). Fork: bolt.diy, MIT.

**Replit Agent**: autonomous full-stack builder; persistence flaky beyond ~30 min sessions per 2026 reviews.

**Cursor Composer / Composer 2** (March 2026): multi-file diff-review model; parallel reads up to 15 files. Diffs are first-class artifacts.

**Devin**: end-to-end project artifacts (PRs, plans, runs) rather than discrete typed objects.

**e2b**: Firecracker microVM sandboxes; OpenAI Agents SDK provider since April 2026; live preview URLs per sandbox; produces "artifacts you can review before publishing."

**Cloudflare Sandboxes / Dynamic Workers** (GA April 2026): V8-isolate sandboxes spin in ms (~100x faster than containers), $0.002/unique-Worker/day plus standard Workers pricing; live preview URLs.

## Taxonomies

A **typed-artifact vs. blob** debate has crystallized. The emerging consensus (Bolt, Claude, v0, MCP) is *typed artifacts*: each output declares a kind (`code/file`, `code/patch`, `web/app`, `doc/markdown`, `doc/office`, `chart/svg`, `data/table`, `dashboard/live`) plus a MIME-ish content type. Research literature (Emergent Mind, "Agentic Artifacts") defines them as **machine-consumable, version-controlled entities** produced in Plan-Act-Observe-Revise loops — distinct from chat messages because they participate in feedback. Examples in literature: chain-of-evidence JSON (TxRay), slide decks (DeepPresenter), labeled-image pairs (ArtiAgent), context files (AGENTS.md/SKILL.md).

The split that matters for paperclip: **patches vs full-file snapshots vs deployable apps vs static documents vs live/bound artifacts**. The last category (Claude Live Artifacts) is the 2026 novelty — artifacts that re-evaluate against bound data sources.

## Versioning

No clear winner, but a pattern: **snapshot-per-revision is dominant for user-facing artifacts** (Claude, Canvas, v0). **Git-style diffs** dominate when artifacts represent code changes against an existing repo (Cursor Composer, Devin, Aider). Bolt streams *append-only action logs* that reduce to filesystem state. CRDTs are absent in mainstream agent stacks (still confined to collaborative editors). Best-practice synthesis: **immutable revision snapshots with parent pointers**, plus a derived diff view, plus an event/action log for replay/audit. This matches paperclip's existing `document_revisions` shape.

## Preview hosting

Three tiers in 2026:

- **V8 isolates (Cloudflare Dynamic Workers)**: sub-ms cold start, MB-scale memory, JS-only, cheapest at scale. Best for static/SPA preview and JS workers.
- **microVMs (e2b/Firecracker, Fly Machines)**: 100-500 ms cold start, full Linux, any runtime, stronger isolation. Best for full-stack/agent code execution. e2b is the de-facto agent sandbox after the OpenAI Agents SDK integration.
- **Platform preview URLs (Vercel, Cloudflare Pages)**: per-deployment, durable, but heavier and tied to a build pipeline.

Tradeoff: isolates win on cost/latency; microVMs win on fidelity and security boundary; platform previews win on persistence and DX. Security: microVMs are the only tier that safely runs untrusted compiled binaries.

## Standards

- **MCP Resources** (spec dated 2025-11-25, 2026 roadmap adds streaming + pagination): URI-addressed text or base64 blobs with custom schemes. The closest thing to a cross-vendor "artifact" primitive but read-only and context-shaped, not produce-shaped. No formal "MCP Artifact" primitive yet.
- **OCI Artifacts / ModelPack** (CNCF Sandbox, June 2025) and **Docker Model Runner** (GA Sept 2025): standardize *model* packaging via OCI manifests with custom media types and content-digest layers. Pattern is reusable for agent work products: manifest + typed layers + annotations.
- **AGENTS.md** (Linux Foundation / Agentic AI Foundation, mid-2025): cross-tool open format adopted by OpenAI Codex, Cursor, Amp, Jules, Factory. Plain Markdown, no schema. Companion: **SKILL.md**.
- No "Open Artifacts" spec yet. JSON Schema-typed tool I/O is the de-facto contract (KnowQL etc.).

## Paperclip integration considerations

Paperclip's existing `heartbeat_runs`, `document_revisions`, `issue_work_products`, and `execution_workspaces` already cover the right primitives — an artifact layer should *unify*, not replace them.

- **Attach points**: artifacts should be polymorphic on `(run_id, issue_id)` — a run produces artifacts, issues aggregate them across runs. Mirrors Devin/Cursor patterns.
- **Typed kinds**: declare an artifact taxonomy (`code.file`, `code.patch`, `doc.markdown`, `doc.office`, `web.app`, `chart`, `data.table`, `live.dashboard`) with a JSON Schema per kind. Borrow OCI's manifest+layers shape: artifact row = manifest, blobs = content-addressed.
- **Versioning**: reuse `document_revisions` semantics (snapshot + parent) for all artifact kinds; derive diffs lazily; keep an action/event log on `heartbeat_runs` for replay.
- **Preview hosting**: pluggable provider — local execution_workspace for dev, e2b/Cloudflare Sandbox for hosted. Plugin system fits perfectly.
- **Declaration**: agents declare artifacts via a typed tool call (MCP-resource-like URI + kind + schema), matching Bolt's `addArtifact` pattern.
- **Lifecycle**: ephemeral preview (TTL'd sandbox) vs durable artifact (content-addressed blob in Postgres/object store); promotion is explicit.

## Open questions

1. Single artifact table vs per-kind tables? OCI-style polymorphic manifest is more extensible but harder to query.
2. Where do live/bound artifacts live — agent-side rebinding vs server-rendered like Claude Live Artifacts?
3. Should paperclip publish an MCP-Resource view of artifacts so external agents can consume them?
4. Diff strategy for non-text artifacts (binary docs, datasets) — content-addressed snapshots only?
5. Plugin contract for preview providers: is it a generic "give me a URL serving this artifact bundle"?
