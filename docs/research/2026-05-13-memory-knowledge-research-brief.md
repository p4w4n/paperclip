# Agent Memory + Knowledge Systems — Research Brief (May 2026)

> Research input that informed `docs/superpowers/specs/2026-05-13-memory-knowledge-design.md`. Generated by web search across 2025-2026 sources.

## Frameworks

- **Mem0** — Memory-as-a-service SDK (Python/TS). Hybrid vector+graph+KV. Apache-2.0. ~47K stars; State-of-Memory 2026 report shipped. Framework-agnostic (LangChain, CrewAI, AutoGen).
- **Letta** (ex-MemGPT) — Stateful agent runtime with OS-style core/recall/archival memory tiers; agents page memory in/out via tool calls. Apache-2.0, Python. Active 2026 release cadence.
- **LangGraph + LangMem** — Memory built on `Store` namespaces (JSON-doc KV) with Postgres/Redis/MongoDB backends; LangMem SDK adds background memory manager + memory tools. TS + Python.
- **LlamaIndex** — Token-buffer + chat-store memory plus `VectorMemoryBlock`/`FactExtractionMemoryBlock`; biased to RAG-first agents. MIT, Python/TS.
- **Zep / Graphiti** — Temporal knowledge-graph memory engine over Neo4j+vector hybrid; bi-temporal validity windows on every fact. Apache-2.0, Python. ~14K stars in 8 months. arXiv 2501.13956 reports +18.5% accuracy / -90% latency vs MemGPT on DMR.
- **CrewAI** — Built-in short-term + long-term + entity memory; pluggable vector store. MIT, Python.
- **AutoGen / AG2** — Custom memory stores; conversational state across multi-agent groups. CC-BY/MIT.
- **Vercel AI SDK** — `experimental_memory`-style helpers; minimal, leans on host DB.
- **MemoryOS** (BAI-Lab, EMNLP 2025) — Hierarchical short/mid/long-term persona memory; ships an MCP server.
- **MemOS** (MemTensor) — "Self-evolving" memory OS, v2.0 Dec 2025, claims 35% token savings; multi-modal + tool memory.
- **A-MEM** (NeurIPS 2025) — Zettelkasten-style self-linking notes; memory evolves as new notes update old ones.

## Taxonomies

The settled axes in 2026: **episodic** (events/turns/tool-calls), **semantic** (consolidated facts), **procedural** (reusable skills/plans, formalized in Mem^p, arXiv 2508.06433). Orthogonal scope axes: **session/run**, **agent**, **user**, **org/app** — Mem0's four-scope model is the de facto standard, with retrieval ranking user > org > agent > session. Working memory (in-context) vs long-term (external store) is universal. Reflection produces semantic from episodic; skill mining produces procedural from episodic.

## Storage

**pgvector 0.9** (HNSW) is competitive with Qdrant/Weaviate up to ~10M vectors and is the default if you already run Postgres; beyond ~10M, Qdrant (Rust, payload-indexed filtered search) and `pgvectorscale` pull ahead — Qdrant ~471 QPS @ 50M @ 99% recall. Weaviate wins for hybrid (BM25+vec) and multi-tenant. **Graph hybrid**: Neo4j+vector via Graphiti is the leading pattern for temporal/relational memory; Memgraph is the lighter alternative. **Embeddings**: `voyage-3-large` is the 2026 MTEB leader (65.1) and ~10% better than OpenAI `text-embedding-3-large` on retrieval; OpenAI remains the safe default; int8 quantization at 1024-dim is now standard for memory stores (8x storage cut, <0.5% quality loss).

## Reflection

Two dominant patterns: (1) **agent-loop reflection** (Letta, A-MEM) — the agent itself decides what to promote/evict via tool calls; (2) **background consolidation** (Mem0, Zep) — async workers extract facts, merge duplicates, summarize old turns, run entity/relation extraction. Selection criteria converging on: salience, novelty vs existing memory, recency-decay, access frequency, and explicit user signal. Zep adds bi-temporal invalidation (fact superseded at T) instead of deletion. Staleness vs correctness remains an open problem (Mem0 SOM-2026).

## Standards

**MCP** (Anthropic, now Linux Foundation AAIF) is the de facto agent-to-tool standard with 18K+ servers; the 2025-11-25 spec covers persistent context layers (user/session/org). MemoryOS-MCP and OpenClaw plugins expose memory backends as MCP servers — this is the closest thing to a portable memory protocol today. Google **A2A** handles agent-to-agent. No dedicated "memory protocol" has won; MCP+namespacing is being used as one. A Q3 2026 MCP/A2A joint spec is expected.

## Paperclip integration considerations

- **Scope mapping**: paperclip's `companies` ~ org scope; `agents` ~ agent scope; `issues`/`runs` ~ session scope; humans/operators ~ user scope. The Mem0 4-scope model maps cleanly; retrieval should union-rank with company memories as ground-truth-ish, agent memories as learned behavior, issue memories as ephemeral context.
- **Storage**: with Postgres already in the stack, pgvector + HNSW for episodic/semantic is the obvious starting point; reserve Graphiti-on-Neo4j as an opt-in plugin for customers needing temporal/relational reasoning.
- **document_revisions** is a natural episodic store; semantic/procedural memory should be a *derived* table populated by a background worker (fits the worker-fleet model).
- **Plugin-extensibility**: expose memory as a runtime service on workers with a stable interface (`recall`, `write`, `forget`, `reflect`) so plugins can swap Mem0/Zep/Letta backends. Ship an MCP-server adapter so external agents can use paperclip memory.
- **Agents-as-issues**: each issue should own a session memory that consolidates into agent memory on close; runs append episodic, reflection runs in worker background.
- **Retention**: per-scope TTL + decay; user-deletable per GDPR; tenant-isolated by company.

## Open questions

- Sync vs async reflection — block run completion or fire-and-forget?
- Cross-issue procedural memory: do skills learned in one issue transfer to siblings under the same agent? Same company?
- Conflict resolution when agent memory contradicts company memory — bi-temporal (Zep) or last-write-wins?
- Embedding choice: lock to OpenAI for simplicity vs pluggable (Voyage for code/legal tenants)?
- Is memory a first-class table in core schema, or always plugin-provided?
- MCP-server surface: expose paperclip memory outward, consume external memory inward, or both?
